# 阶段零：前置必要改造 —— 核心骨架大换血

> **核心目标**：彻底丢掉局部的硬编码循环，让系统变成纯粹的“状态机”，这是系统从散装的“脚本”转变为真正的 **Agentic Workflow 工程化项目**的绝对基石。如果不做这步剥离，后续的高级图打断、对接 Web UI 等进阶特性将面临重大的返工。

以下是阶段 0 的 5 个具体改造项剖析（过去 vs 现在）：

## 1. 重构 State (全局状态定义)
这是系统转向状态机的第一步，让所有节点（Agent）拥有统一的“大脑”。

*   **原来是什么**：在旧版中，各个 Agent（如面试助手、学习助手）各自为战，将聊天记录存放在自己的独立变量或列表中。它们彼此信息隔离，全局数据不同步。
*   **现在是什么**：完全抽离出一个全局唯一的 `CareerAssistantState`。无论是学习、面试还是简历修改，所有的历史消息、上下文都存放于这个中心化的 State 对象中，任何节点修改 State 都会对全图可见。
*   **使用组件**：使用 `TypedDict` 定义结构，并使用 LangGraph 的 `add_messages` 作为 Reducer 聚合器，自动管理消息队列的合并。
    ```python
    from typing import TypedDict, Annotated
    from langgraph.graph.message import add_messages
    
    class CareerAssistantState(TypedDict):
        messages: Annotated[list, add_messages]
    ```

## 2. 拆除内部阻塞 (移除 `while True` 和 `input`)
旨在将 Agent 从一个阻塞程序降维成纯粹的“处理节点”。

*   **原来是什么**：在 `learning_agent.py` 或 `interview_agent.py` 中存在基于 Python 的 `while True:` 循环，并在其中调用命令行 `input()` 等待输入。这锁死了整个执行流，导致用户卡在一个频道的死循环中无法脱身。
*   **现在是什么**：强制砸碎死循环，删除底层的 `input()`。所有 Agent 节点全部转化为**纯函数（Pure Function）**架构：接收当前 `State` -> 触发 LLM 思考或调用工具 -> 向 `State` 吐出新结果 -> 函数随之结束，并将总控制权交还给母图。
*   **使用组件**：将原循环重构为 LangGraph 的标准 `Node`。当内部仍需要多步工具调用时，调用原生的 `create_tool_calling_agent` 和 `AgentExecutor`，依靠执行器内生的逻辑在单次节点运转中做到闭环。

## 3. 中心化路由 (重塑 Supervisor 机制)
解决用户意图跳跃漂移的问题。

*   **原来是什么**：系统仅在第一句话初入时判断一次意图，随后便一头扎进特定助手的循环深处。若用户中途跨行发问（例如面试中途询问理论概念），系统缺乏抽身调度的能力。
*   **现在是什么**：拆掉内循环后，设立了一个高级调度中心：`categorize` 节点。图的每一次流转，用户的**每一句话**，都会先由该前置路由重新评估意图，敏锐捕捉话题突变，实现“学习频道”与“面试频道”之间的无缝横跳机制。
*   **使用组件**：高度依赖 LangGraph 的**条件边 (`add_conditional_edges`)** 功能。配合大型语言模型基于 `with_structured_output()` 强制输出枚举类型字符串（例如 "learning", "interview", "general"），以此充当控制岔路的“指示牌”。

## 4. 引入持久化机制 (接入 Checkpointer)
为 Graph 赋予跨执行周期的长期绝对记忆能力。

*   **原来是什么**：脚本式的程序一旦关闭终端重启，或因异常崩溃退出，存放在内存里的所有历史对话瞬间清空，只能重开。
*   **现在是什么**：在编译（Compile）有向无环图（DAG）的时候，为大图注射存档器。用户的每一个节点交互回合，系统都会按照唯一的 `thread_id` (会话ID) 和时间戳保存执行状态快照。下次传入同一 ID，Agent 即可跨越时空连贯上下文。
*   **使用组件**：使用 `langgraph.checkpoint.memory` 中的 `MemorySaver`（在真实的生产级项目中可轻松替换为 `SqliteSaver` 甚至 `PostgresSaver`）。
    ```python
    from langgraph.checkpoint.memory import MemorySaver
    
    memory = MemorySaver()
    app = workflow.compile(checkpointer=memory) # 给图挂载记忆体
    ```

## 5. 顶层驱动 (解耦 I/O 交互)
为后续对接 Web 前端或 FastAPI 接口扫清障碍。

*   **原来是什么**：接收键盘字符串的 `input()` 和吐出结果的 `print()` 深深嵌入、散落在各个业务逻辑层深处。算法与终端 I/O 混在一起。
*   **现在是什么**：把内部所有的 I/O 剔除后，整个大图系统变成了一个结构非常干净的计算黑盒。那唯一的一个需要人类敲键盘互动的 `input()`，被提取到且仅保留在全局最顶层的 `main.py` 驱动文件引擎里。
*   **使用组件**：顶层通过传参的形式，以 `app.invoke()` 甚至 `app.stream()` 来唤醒图流转。
    ```python
    # 通过配置字典传入唯一会话 ID
    config = {"configurable": {"thread_id": "session_001"}}
    # 模拟外部输入向大图注水
    response = app.invoke({"messages": [("user", user_input)]}, config)
    ```
