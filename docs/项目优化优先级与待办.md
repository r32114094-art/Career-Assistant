# GenAI Career Assistant 面试导向优先级与待办

更新时间：2026-04-18

## 文档目标（面试导向）

这份待办不以“最快上线”为核心，而以“体现工程思维能力”为核心。  
面试时要让对方看到你具备：

- 问题定义能力：能把“路由不准”拆成可测子问题。
- 方案取舍能力：能解释为何选这个架构而不是另一个。
- 实验闭环能力：有 baseline、有改进、有量化结果、有失败复盘。
- 工程落地能力：有可复现实验、可维护代码、可讲清的技术决策。

---

## 优先级定义（重新排序）

- P0：面试核心亮点（必须完成，优先展示“思维深度”）
- P1：工程深度加分项（完成后显著拉开和“课程作业项目”的差距）
- P2：基础卫生项（不做会扣分，做了是保底）

---

## P0（面试核心亮点，先做）

### 1. 路由系统从“字符串解析”升级到“结构化决策”

目标：把路由从脆弱规则升级为可解释、可扩展的决策层。

详细实施计划：
- [ ] 见 `docs/P0-1_结构化路由改进实施计划.md`

涉及文件：
- `nodes/categorize.py`
- `router.py`
- `eval/eval_routing.py`

待办：
- [ ] 分类输出改为结构化格式（例如 `main_intent/sub_intent/confidence/reason`）。
- [ ] 路由逻辑改为字段决策，移除正则和关键词兜底逻辑。
- [ ] 为低置信度样本增加澄清分支（而不是硬路由）。

验收标准：
- 路由输入输出均可严格解析（无“模型随意文本”依赖）。
- 对混合意图和短跟随语有稳定策略，不再随机漂移。

面试可讲亮点：
- “我把 prompt 工程问题转成了可验证的状态机问题。”

---

### 2. 做一套“可复现实验”而不是只报准确率

目标：展示你是用实验驱动迭代，而不是拍脑袋改 prompt。

涉及文件：
- `eval/eval_routing.py`
- `eval/routing_dataset.json`
- `路由测试.md`

待办：
- [ ] 固定 baseline（当前版本）与改进版（结构化路由）双跑脚本。
- [ ] 输出统一报告：准确率、F1、混淆矩阵、误差分桶、耗时分布。
- [ ] 建立误差标签体系：混合意图、边界语义、上下文漂移、格式异常。

验收标准：
- 同一数据集可重复复现实验结果（同配置下波动可解释）。
- 结果报告能回答“为什么提升/为什么没提升”。

面试可讲亮点：
- “我不仅做了优化，还做了误差归因和实验可复现。”

---

### 3. 建立“失败驱动迭代”证据链（关键）

目标：让面试官看到你会系统性解决问题，而非只贴成功结果。

涉及产物：
- `docs/` 下新增复盘文档（建议：`routing_error_postmortem.md`）

待办：
- [ ] 挑 3-5 个典型误判案例，写清：现象 -> 假设 -> 改动 -> 结果。
- [ ] 明确哪些改动无效、为什么无效（必须保留“失败尝试”）。
- [ ] 总结决策原则（何时分类、何时澄清、何时拒答）。

验收标准：
- 至少有 1 条“失败改动”被保留并解释。
- 面试时能完整讲一条 STAR 故事（Situation/Task/Action/Result）。

面试可讲亮点：
- “我记录了失败路径，证明我有迭代方法论。”

---

### 4. 输出 1 份架构决策记录（ADR）

目标：体现架构取舍能力，不让项目看起来像拼装代码。

涉及产物：
- `docs/ADR-001-routing-architecture.md`（建议）

待办：
- [ ] 记录候选方案（纯 Prompt 分类 / 规则路由 / 结构化分类 + 状态机）。
- [ ] 写明选择依据（准确率、鲁棒性、维护成本、可解释性）。
- [ ] 写明已知 trade-off（例如延迟增加、实现复杂度上升）。

验收标准：
- ADR 可独立阅读，评审者能理解“为什么这么设计”。

面试可讲亮点：
- “我做了架构方案比较，不是只会堆功能。”

---

## P1（工程深度加分）

### 5. 引入 Prompt/路由版本管理（PromptOps）

目标：把 prompt 改动纳入工程流程，而不是散落在代码里。

涉及文件：
- `nodes/categorize.py`
- `docs/`（新增 changelog）

待办：
- [ ] 将关键 prompt 模板抽离并版本化。
- [ ] 建立 `prompt_version -> eval_result` 映射记录。
- [ ] 每次改动必须附实验结果和风险说明。

验收标准：
- 任意一次准确率波动都能追溯到具体 prompt 版本。

---

### 6. 上下文管理做成可配置策略

目标：展示你对长对话质量/成本权衡有系统方案。

涉及文件：
- `utils.py`
- `state.py`
- `nodes/*.py`

待办：
- [ ] 把历史裁剪从固定条数升级为 token budget + 关键信息保留。
- [ ] 做 A/B 对比：短上下文 vs 长上下文 vs 摘要上下文。
- [ ] 报告“质量-延迟-成本”三角取舍。

验收标准：
- 有一张对比表可以解释为何最终选某策略。

---

### 7. 工具层抽象与回退机制

目标：体现系统设计能力，而非绑定单一第三方工具。

涉及文件：
- `agents/*.py`
- `requirements.txt`

待办：
- [ ] 抽象统一 `SearchProvider` 接口。
- [ ] 支持主备搜索源（例如 SerpAPI 不可用时自动降级）。
- [ ] 增加失败重试和错误分类（配置缺失/限流/网络故障）。

验收标准：
- 主搜索不可用时仍能给出可解释退化行为。

---

### 8. 可观测性最小闭环（面试加分很明显）

目标：让你能量化讲“哪个节点慢、哪个节点错”。

涉及文件：
- `app.py`
- `workflow.py`
- `docs/`（新增指标说明）

待办：
- [ ] 统一 request_id，串联一次请求的全链路日志。
- [ ] 记录节点耗时、错误率、重试次数。
- [ ] 输出简版监控报表（可脚本生成 CSV/Markdown）。

验收标准：
- 可以回答“性能瓶颈在哪一层、为什么”。

## P2（基础卫生，作为保底）

### 9. 安全和协议一致性收口

目标：避免明显工程漏洞被面试官追问扣分。

涉及文件：
- `static/index.html`
- `static/app.js`
- `app.py`

待办：
- [ ] Markdown 渲染增加 sanitize（XSS 防护）。
- [ ] WebSocket 会话增加最小鉴权机制。
- [ ] HITL 前后端协议字段对齐（支持自定义修改意见）。

验收标准：
- 基础安全问题不过于显眼。
- 协议行为一致、可解释。

---

### 10. 项目叙事包装（不是美化，是可讲性）

目标：让项目经历在 5 分钟内讲得“有结构、有证据”。

涉及文件：
- `README.md`
- `docs/`（新增面试讲解稿）

待办：
- [ ] README 增加“问题定义 -> 架构 -> 实验 -> 结果 -> 复盘”结构。
- [ ] 准备 1 页面试讲解提纲（含关键图和关键数字）。
- [ ] 准备 3 个高频追问回答（如“为什么不用纯规则？”）。

验收标准：
- 不看代码也能让面试官理解你的工程贡献和思考过程。
