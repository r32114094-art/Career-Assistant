# P0-1 结构化路由改进实施计划

更新时间：2026-04-18
适用范围：`nodes/categorize.py`、`router.py`、`state.py`、`workflow.py`、`eval/eval_routing.py`

## 1. 背景与问题定义

当前路由的核心问题是“模型自由文本输出 + 代码猜测解析”：

- 主分类靠正则抓数字（如 `re.search(r"[1-5]")`）。
- 子分类靠关键词包含判断（如 `"tutorial" in category` / `"mock" in category`）。
- 评测脚本也用同类字符串抽取方式，导致线上逻辑与评测逻辑一起脆弱。

这会带来三个工程问题：

- 可解释性弱：很难明确说明“为什么路由到这个分支”。
- 稳定性弱：模型输出格式一漂移，路由就可能误判。
- 可演进性弱：后续新增意图、加澄清策略时改动成本高。

---

## 2. 改造目标（面试可讲版本）

把路由系统升级为“结构化决策层”，核心目标如下：

- 目标A：决策契约化。分类节点输出固定 schema，而不是自由文本。
- 目标B：路由确定化。路由器按字段判断，不再猜测字符串。
- 目标C：低置信度可控。增加澄清分支，减少硬路由误判。
- 目标D：评测一致化。评测口径与线上逻辑完全一致，可重复复现。

---

## 3. 分阶段实施计划（每一步含原因与目的）

### 阶段 0：建立基线与保护网（先量化，再改造）

动作：
- 0.1 固定当前 baseline 指标（主分类准确率、各类 F1、误差样本清单）。
- 0.2 备份当前评测输出到 `eval/results/baseline_*`。
- 0.3 为 `router.py` 关键路径补 3-5 个最小回归用例（混合意图/短跟随语/上下文切换）。

原因：
- 没有基线就无法证明改造有效，面试时也无法回答“你到底提升了什么”。

目的：
- 建立“可比较”的起点，避免改造后只有主观感受没有证据。

产出：
- 一份 baseline 报告。
- 一组可重复执行的最小回归测试。

---

### 阶段 1：定义路由决策 Schema（先定协议，再写代码）

动作：
- 1.1 在 `state.py` 增加结构化字段 `routing_decision`（TypedDict 或 dataclass 风格）。
- 1.2 定义字段枚举：
  - `main_intent`: `learning|resume|interview|job_search|out_of_scope`
  - `sub_intent`: `question|tutorial|mock|null`
  - `confidence`: `0.0~1.0`
  - `needs_clarification`: `true|false`
  - `clarify_question`: `string|null`
  - `reason`: `string`
- 1.3 在 `docs/` 产出一页“决策字段字典”，写明每个字段语义和边界。

原因：
- 先定义契约，后续节点和评测才能对齐；否则容易边改边漂移。

目的：
- 让“模型输出 -> 路由执行 -> 评测统计”有统一语言，降低歧义。

产出：
- 稳定 schema（可被代码、日志、评测共同消费）。

---

### 阶段 2：改造分类节点输出（自由文本 -> 结构化对象）

动作：
- 2.1 改 `nodes/categorize.py`：要求模型输出严格 JSON（仅包含 schema 字段）。
- 2.2 增加输出解析与校验：
  - JSON 解析失败时，走受控兜底（记录解析错误 + 回退策略）。
  - 枚举值不合法时，统一映射到 `out_of_scope` 或 `needs_clarification=true`。
- 2.3 过渡期双写：
  - 新字段写入 `routing_decision`
  - 旧字段 `category` 暂时保留，避免一次性改动过大。

原因：
- 一步全量切换风险高；双写可减少回归风险并便于对比新旧路径差异。

目的：
- 先把“生成层”变稳定，为后续“路由层”改造铺路。

产出：
- 分类节点稳定输出结构化决策。

---

### 阶段 3：改造路由执行逻辑（猜字符串 -> 读字段）

动作：
- 3.1 `route_query` 仅读取 `routing_decision.main_intent`，移除正则解析。
- 3.2 `route_learning` / `route_interview` 改为读取 `sub_intent`。
- 3.3 增加澄清路由：
  - 若 `needs_clarification=true` 或 `confidence` 低于阈值（如 0.65），转到 `clarify` 节点。
- 3.4 在 `workflow.py` 新增 `clarify` 节点和对应条件边。

原因：
- 只要继续依赖字符串解析，格式漂移问题就不会真正消失。

目的：
- 让路由成为“确定性状态机”，并在不确定时主动澄清而非硬判。

产出：
- 可解释、可扩展、可控的路由分发层。

---

### 阶段 4：补齐澄清节点（把“不确定”变成可管理流程）

动作：
- 4.1 新增 `nodes/clarify.py`，统一输出澄清问题（如“你是想学习还是找工作？”）。
- 4.2 约束澄清轮次上限（例如 1-2 轮），避免无限追问。
- 4.3 澄清后回流到分类节点，形成闭环。

原因：
- 混合意图、短跟随语是当前主要误判来源，澄清是比强行分类更稳的策略。

目的：
- 用“交互确认”替代“猜测判断”，提高最终成功率和用户体验一致性。

产出：
- 低置信度样本的受控处理机制。

---

### 阶段 5：评测体系同步升级（确保线上与离线口径一致）

动作：
- 5.1 改 `eval/eval_routing.py`：直接读取 `routing_decision` 字段评测，不再正则抽取。
- 5.2 新增指标：
  - 结构化解析成功率
  - 低置信度触发率
  - 澄清后路由正确率
  - 端到端正确率（含澄清流程）
- 5.3 新增误差分桶报表（格式错误、语义冲突、阈值策略问题）。

原因：
- 如果评测脚本仍是旧逻辑，就无法证明新架构真实有效。

目的：
- 构建“线上逻辑 = 评测逻辑”的证据闭环。

产出：
- 改造前后可直接对比的统一报告。

---

### 阶段 6：灰度与回滚策略（工程化落地）

动作：
- 6.1 增加开关：`ROUTING_MODE=legacy|structured`。
- 6.2 先在本地和小流量会话使用 `structured`，观察错误类型。
- 6.3 保留快速回滚路径（开关切回 `legacy`）。

原因：
- 结构化改造涉及核心入口，必须有安全回退，不可“孤注一掷”。

目的：
- 在保证稳定性的同时推进升级，体现工程风险控制能力。

产出：
- 可灰度、可回滚的上线策略。

---

## 4. 预期效果（建议按区间表达）

说明：具体收益需以你项目实测为准，以下是合理目标区间。

- 主分类准确率：预计提升 `+2 ~ +5` 个百分点。
- 学习类 Recall：预计提升 `+10 ~ +20` 个百分点（尤其是混合意图/短跟随语）。
- 误判类型分布：格式漂移导致的误判显著下降。
- 系统可解释性：每次路由都有字段化 `reason` 与 `confidence` 可追踪。

面试价值提升：
- 可以讲“状态机 + schema + 低置信度澄清 + 灰度回滚 + 指标闭环”，
- 明显区别于“只调 prompt 的 demo 项目”。

---

## 5. 风险与应对

- 风险1：模型偶发不按 JSON 输出。
  - 应对：强校验 + 重试一次 + 安全兜底分支。

- 风险2：置信度阈值设置不当导致澄清过多或过少。
  - 应对：离线网格搜索阈值（0.55/0.65/0.75）并比较端到端指标。

- 风险3：改动面广引发回归。
  - 应对：双写过渡 + 开关灰度 + 可回滚。
